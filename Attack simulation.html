Adversarial Demo · Vishing

When the attack
is a panicked user

Social engineering doesn't exploit technical vulnerabilities — it exploits cognitive ones. This demo simulates a real vishing attack and shows how the current iToken system fails, and how the behavioral redesign holds.

VISHING · Fake Itaú support call · Wire R$8,500 → attacker's account
STAGE 01 · ATTACK
The call
A fake "Itaú Security" agent makes contact. The victim receives an urgent message: account compromised, unauthorized transfer in progress, immediate action required.
STAGE 02 · FAILURE
iToken capitulates
The current system applies identical authentication to all operations. The user, in a state of induced urgency, confirms the token without processing what they are approving.
STAGE 03 · DEFENSE
Redesign intervenes
The behavioral system detects anomalies (new payee, high value, atypical time), imposes a deliberate pause and out-of-band verification — breaking the urgency state before it can be exploited.

The call comes in

7:14 PM. Lucas receives a call from a local number. The voice is professional, urgent, and uses real personal data leaked from a prior breach to appear legitimate.

CRITICAL FAILURE: uniform friction. R$8,500 to a new payee receives the exact same confirmation flow as a R$20 Pix to a frequent contact. The "0 transactions" field is visible but not flagged as a risk signal.

PSYCHOLOGY: Lucas is in System 1 mode (urgency). He does not read the details — he executes the attacker's script. The 6-digit token requires no semantic processing of the operation being approved.

OUTCOME: R$8,500 transferred. The attacker hangs up. Lucas only discovers the fraud when he checks his statement — the operation was technically legitimate. The system worked exactly as designed.
🔴
iToken: attack successful
Uniform authentication cannot distinguish a legitimate operation from one performed under cognitive coercion. The system was designed to verify identity — not to protect the decision.

WHY IT WORKED: the 47s pause forced a transition to System 2 (deliberate processing). The notification on the second device created an independent verification point the attacker did not control. Artificial urgency was neutralized by forced time.
⏱ Deliberate pause 📲 Out-of-band verification ⚖ Friction proportional to risk 🔍 Risk transparency
✓
Redesign: attack neutralized
Not through stronger cryptography. Not through more complex authentication. Through a 47-second behavioral interruption that restored the user's ability to deliberately process what they were approving.

Stage 04 · Analysis

What the demo demonstrates

The difference between the two systems is not technical — it is epistemic. One was designed to verify identity. The other was designed to protect decisions.

🎯
Uniform friction is a vulnerability
When every operation receives the same level of authentication, users learn to confirm without processing. Habituation eliminates the informational value of the token. iToken does not distinguish risk — and so cannot raise the alarm when it matters.
⏱
Deliberate pause as defense
47 non-bypassable seconds are not arbitrary friction — they are a cognitive intervention. The urgency induced by the attacker is time-based. Forcing processing time neutralizes the primary technique of vishing without adding complexity for the legitimate user.
📲
Out-of-band verification
The attacker controls the voice channel. But not the second device. The push notification creates an independent verification point — and unlike a numeric token, it requires the user to read and understand the operation before proceeding.
📄
The paper's central argument
This demo is an interactive demonstration of the thesis in Carvalho (2026): authentication failures in banking systems are primarily behavioral, not technical. iToken solves the wrong problem — it verifies who the user is, but does not protect the integrity of the user's decision. The proposed redesign addresses the layer that matters: the user's cognitive state at the moment of approval.
